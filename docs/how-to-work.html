Інструкція

Як працювати з prompt_party

Інструкція для студентів і викладача: як використовувати навчальний репозиторій, виконувати завдання, створювати власні проєкти та публікувати результати.

Що таке prompt_party

prompt_party — це навчальний репозиторій і сайт з практичними завданнями для студентів. Він використовується для навчання роботі з Git, GitHub, GitHub Pages, Codex CLI, Excel/CSV-даними та простими AI-автоматизаціями.

Головний репозиторій:
https://github.com/StsZu/prompt_party

Навчальний сайт:
https://stszu.github.io/prompt_party/

1. Для чого потрібен prompt_party

Репозиторій prompt_party — це головний навчальний портал.

У ньому зберігаються:

  • практичні завдання для студентів;
  • HTML-сторінки з інструкціями;
  • приклади CSV/Excel-файлів;
  • HTML-презентації;
  • посилання на студентські роботи;
  • матеріали для навчання Git, GitHub і Codex CLI.

Важливо: prompt_party — це не місце, де кожен студент напряму змінює головний сайт. Це центральний навчальний репозиторій. Студенти виконують завдання у власних репозиторіях, а в prompt_party можуть додаватися тільки посилання на готові роботи або окремі матеріали після перевірки.

2. Як студент працює із завданням

Студент відкриває навчальний сайт: 

https://stszu.github.io/prompt_party/

Потім обирає практичне завдання, наприклад: 

Finance Excel Checker

Після цього студент виконує завдання у власному локальному проєкті на своєму комп’ютері.

Приклад власного проєкту студента:

finance-excel-checker/
├── payments.csv
├── check_payments.py
├── README.md
└── .gitignore

Студент повинен:

  1. створити локальну папку проєкту;
  2. ініціалізувати Git;
  3. додати CSV-файл з фінансовими даними;
  4. зробити перший commit;
  5. створити окрему Git-гілку;
  6. запустити Codex CLI;
  7. створити Python-скрипт для перевірки фінансових даних;
  8. перевірити результат;
  9. оформити README;
  10. запушити роботу на GitHub.

3. Де студент зберігає свою роботу

Кожен студент створює власний GitHub-репозиторій.

Наприклад:

https://github.com/student-name/finance-excel-checker

У цьому репозиторії студент зберігає свою практичну роботу. Основний репозиторій prompt_party не треба копіювати і не треба змінювати напряму для виконання першого завдання.

4. Що студент здає

Після виконання завдання студент здає:

  • посилання на GitHub-репозиторій;
  • посилання на GitHub Pages-презентацію, якщо вона створена;
  • скріншот запуску Python-скрипта;
  • короткий опис, що було зроблено.

Приклад:

Студент: Олена Шуть
Проєкт: Finance Excel Checker
GitHub: https://github.com/username/finance-excel-checker
Сайт: https://username.github.io/finance-excel-checker/
Опис: створено CSV-файл з платежами, Python-скрипт перевірки помилок, README та історію комітів.

5. Як студент може зробити HTML-презентацію

Після виконання основного завдання студент може створити HTML-презентацію у своєму власному репозиторії.

Для цього в репозиторії студента можна створити папку:

docs/
├── index.html
└── style.css

Потім у GitHub треба увімкнути GitHub Pages: 

Settings → Pages → Deploy from a branch → main → /docs

Після цього студент отримає власний сайт-презентацію: 

https://username.github.io/finance-excel-checker/

На цій HTML-сторінці студент може коротко показати:

  • назву проєкту;
  • мету роботи;
  • структуру файлів;
  • що робить Python-скрипт;
  • приклади знайдених помилок;
  • посилання на GitHub-репозиторій;
  • висновок.

6. Чи треба студентам додавати свої презентації прямо в prompt_party

На першому етапі — ні.

Правильна модель така:

prompt_party = головний навчальний портал
student repo = робота конкретного студента
student GitHub Pages = презентація конкретного студента

У prompt_party краще додавати не всі файли студентів, а тільки посилання на їхні роботи.

Наприклад:

Олена Шуть
Проєкт: Finance Excel Checker
GitHub: https://github.com/username/finance-excel-checker
Сайт: https://username.github.io/finance-excel-checker/

Так головний сайт залишиться чистим і зрозумілим.

7. Рекомендований порядок навчання

Етап 1. Git і GitHub

Студент створює локальний проєкт, робить commit і push у власний GitHub-репозиторій.

Етап 2. Codex CLI

Студент запускає Codex CLI у своєму проєкті та просить створити або покращити файли.

Етап 3. GitHub Pages

Студент створює просту HTML-презентацію і публікує її через GitHub Pages.

Етап 4. Каталог робіт

Викладач додає посилання на студентські роботи в prompt_party.

8. Правило безпеки

Не треба працювати напряму в гілці main, якщо зміни робить Codex CLI або інший AI-інструмент.

Краще створити окрему гілку:

git switch -c add-payment-validation

Після перевірки змін можна зробити commit і push. Це допомагає не зламати основну версію проєкту.

9. Коротко

Студент:

читає завдання → створює свій проєкт → працює з Git/GitHub → використовує Codex CLI → робить презентацію → здає посилання

Викладач:

публікує завдання → перевіряє роботи → додає посилання на найкращі результати в каталог

Головна ідея: не складати всі студентські файли в один репозиторій, а використовувати prompt_party як чисту навігаційну платформу для завдань, прикладів і результатів.
